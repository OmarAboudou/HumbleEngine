Objectif de cette page

La première approche de Godot présentait le moteur comme une machine qui organise des objets, les met à jour et les affiche. Ici, on commence à regarder Godot avec une vision plus proche de l’architecture moteur.

Les concepts centraux sont : le SceneTree, les Node, le cycle de vie, les scènes, les signaux, les transformations et la séparation entre les nœuds haut niveau et les serveurs bas niveau.

Le cœur de Godot : le SceneTree

Le centre de Godot est un arbre de nœuds. Chaque objet important du jeu est généralement un Node placé dans une hiérarchie. 

Player
├── Sprite2D
├── CollisionShape2D
├── Camera2D
└── AnimationPlayer

Le SceneTree est l’objet global qui possède la racine de cet arbre. Il sert à parcourir les nœuds, appeler leurs callbacks, distribuer les inputs, gérer les scènes, les groupes et les signaux.

Ce qu’est réellement un Node

Un Node est d’abord un objet hiérarchique : il connaît son parent, ses enfants, son cycle de vie et son appartenance éventuelle au SceneTree. 

class Node
{
    Node Parent;
    List<Node> Children;

    virtual void _EnterTree() {}
    virtual void _Ready() {}
    virtual void _Process(float delta) {}
    virtual void _ExitTree() {}
}

Cette idée est importante pour HumbleEngine : avant de faire du rendu ou de l’UI, il faut une base solide pour organiser les objets.

Le cycle de vie des nœuds

_EnterTree

Le nœud entre dans le SceneTree. Le parent existe déjà, mais les enfants ne sont pas encore forcément prêts.

Parent
→ Child
→ PetitEnfant

_Ready

Tous les enfants sont prêts. L’ordre est inversé pour permettre au parent d’utiliser ses enfants en sécurité.

PetitEnfant
→ Child
→ Parent

_Process(delta)

Appelé à chaque frame. Le delta représente le temps écoulé depuis la frame précédente.

while (running)
{
    delta = computeDeltaTime();
    root.Process(delta);
}

_Process vs _PhysicsProcess

Godot sépare la boucle visuelle et la boucle physique.

_Process

Boucle variable selon les FPS. Elle est adaptée à l’animation visuelle, aux effets et à la logique qui dépend directement de l’affichage.

_PhysicsProcess

Boucle fixe, souvent à 60 appels par seconde. Elle est adaptée à la physique, aux collisions et aux comportements qui doivent rester stables.

Dans la nouvelle orientation UI de HumbleEngine, la boucle physique n’est pas prioritaire. En revanche, comprendre la différence reste utile pour plus tard.

Node, Node2D et Control

Node

Objet logique pur, sans position. Il sert pour les managers, la logique, le réseau ou des systèmes abstraits.

Node2D

Ajoute une transform 2D : position, rotation et scale. Il est utilisé pour les objets placés dans un monde 2D.

Control

Base des interfaces utilisateur. Il ajoute anchors, margins, layout, clipping et gestion spécifique de l’input UI.

Pour HumbleEngine, Control devient plus important que Node2D, car le projet se concentre maintenant sur un moteur UI interactif.

Les transformations hiérarchiques

Chaque objet spatial possède une transform locale. Sa transform globale dépend de ses parents. 

Player position = (100, 50)
Sword position = (20, 0)

Sword global position = (120, 50)

Le principe général est : 

Global = Parent * Local

Dans une UI, on retrouve une idée proche : la position finale d’un contrôle dépend de son parent, de son layout, de ses marges et de ses ancres.

Les scènes Godot

Une scène Godot est essentiellement un sous-arbre sérialisé. 

Player
├── Sprite2D
├── Camera2D
└── CollisionShape2D

Quand Godot charge une scène, il lit le fichier, recrée les nœuds, reconnecte les relations parent/enfant et ajoute le tout au SceneTree.

Pour HumbleEngine, cela prépare le futur système de sérialisation de scènes UI.

Les signaux

Les signaux sont un système d’événements. Un nœud peut émettre un événement sans connaître directement ceux qui vont y réagir. 

signal died
emit_signal("died")

Ce mécanisme réduit le couplage entre les systèmes. Pour un moteur UI, il est particulièrement utile pour les boutons, menus, champs de texte et panneaux interactifs.

Le rendu dans Godot

Les nœuds ne dessinent pas directement au GPU. Ils produisent plutôt des intentions ou commandes de rendu. 

Draw texture X
at transform Y
with material Z

Le pipeline simplifié est : parcours du SceneTree, génération des commandes, tri et batching, puis envoi au GPU.

Les serveurs Godot

Godot sépare les systèmes haut niveau et bas niveau avec des serveurs spécialisés.

  • RenderingServer pour le rendu
  • PhysicsServer pour la physique
  • AudioServer pour l’audio
  • NavigationServer pour la navigation

Cette séparation permet aux nœuds de rester proches de la logique utilisateur, tandis que les serveurs gèrent les détails bas niveau.

Conséquence pour HumbleEngine

Le projet s’oriente désormais vers un moteur UI interactif inspiré de Godot. Les systèmes prioritaires deviennent :

  • SceneTree
  • Control
  • layout
  • rendu UI
  • input souris/clavier
  • événements et signaux
  • sérialisation de scènes UI

Les systèmes comme la caméra de jeu, la physique, le son spatialisé et le gameplay temps réel pourront être ajoutés plus tard, une fois la base UI stabilisée.
