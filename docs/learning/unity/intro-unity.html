Trace — Introduction vulgarisée au fonctionnement de Unity

Une vision simplifiée de l’architecture de Unity afin de comprendre comment un moteur de jeu orienté composants peut être construit en C#.

Vision globale

Unity peut être vu comme une grande machine qui organise des objets, leur ajoute des comportements, les met à jour et les affiche à l’écran.

L’idée la plus importante est simple : dans Unity, presque tout est un GameObject auquel on ajoute des Components. 

Unity
├── Éditeur
│   ├── Placer des objets
│   ├── Régler des propriétés
│   ├── Importer des ressources
│   └── Lancer/tester le jeu
└── Runtime
    ├── Game loop
    ├── Scripts C#
    ├── Rendu
    ├── Physique
    ├── Audio
    ├── UI
    └── Animation

GameObject

Un GameObject est une chose présente dans une scène : joueur, caméra, lumière, ennemi, décor, bouton d’interface, projectile, etc.

Seul, un GameObject est presque vide. Il sert surtout de support auquel on accroche des composants. 

Player

À ce stade, Player existe, mais il ne sait pas encore s’afficher, bouger, tomber ou détecter des collisions.

Components

Les Components donnent des capacités au GameObject. Unity utilise donc surtout la composition : on construit un objet en assemblant des briques.

ComponentRôle simplifié
TransformPosition, rotation et taille de l’objet.
SpriteRendererAffiche une image 2D.
Rigidbody2DPermet à la physique de déplacer l’objet.
Collider2DDéfinit la zone de collision.
AudioSourceJoue un son.
Script C#Ajoute une logique personnalisée.
 
Player
├── Transform
├── SpriteRenderer
├── Rigidbody2D
├── BoxCollider2D
└── PlayerController.cs

Ce modèle est important pour HumbleEngine : au lieu de créer une énorme hiérarchie d’héritage, on peut créer un objet simple puis lui ajouter des composants spécialisés.

Scene

Une Scene est une boîte qui contient les GameObjects d’un niveau, d’un menu ou d’un écran de jeu. 

Scene
├── Main Camera
├── Player
├── Enemy
├── Ground
├── Light
└── Canvas

La scène sauvegarde aussi les composants, les propriétés, la hiérarchie et les références entre objets.

Transform et hiérarchie

Chaque GameObject possède un Transform. Il indique où est l’objet, comment il est tourné et quelle est sa taille. 

Player
└── Sword
    └── Hitbox

Si le joueur bouge, l’épée suit. Cela vient du système de transformations parent/enfant : l’enfant possède une position locale relative au parent.

La boucle de jeu

Unity tourne en boucle tant que le jeu est lancé. À chaque frame, il lit les entrées, met à jour les scripts, simule la physique puis dessine l’image. 

while (running)
{
    Input();
    UpdateScripts();
    Physics();
    Animation();
    Render();
    Present();
}

Les scripts C# héritent souvent de MonoBehaviour et Unity appelle automatiquement certaines méthodes. 

public class PlayerController : MonoBehaviour
{
    void Start()
    {
        // Appelé au début
    }

    void Update()
    {
        // Appelé chaque frame
    }
}

Renderer

Le renderer transforme les objets du monde en pixels visibles à l’écran. 

GameObjects
    ↓
Components de rendu
    ↓
Caméra
    ↓
GPU
    ↓
Image affichée

Dans un moteur 2D simple en C#, cela correspondra d’abord à récupérer les sprites visibles, calculer leur position à l’écran, envoyer les données au GPU et afficher l’image finale.

Physique

Le système physique calcule les vitesses, la gravité, les collisions, les forces et les rebonds. 

Rigidbody2D + Collider2D
        ↓
Simulation physique
        ↓
Nouvelle position

La physique est un système séparé du rendu : un objet peut être simulé physiquement avant même d’être dessiné.

Assets et Prefabs

Les assets sont les ressources du jeu : images, sons, modèles, shaders, scènes, scripts, etc.

Un Prefab est un modèle réutilisable de GameObject. Par exemple, un ennemi complet peut être sauvegardé une fois puis instancié plusieurs fois. 

Enemy.prefab
├── Transform
├── SpriteRenderer
├── Rigidbody2D
├── Collider2D
└── EnemyAI.cs

Pour HumbleEngine, cette idée mène directement à un système de ressources et de sérialisation.

Éditeur et runtime

Unity est à la fois un éditeur et un runtime. L’éditeur sert à fabriquer les données du jeu, le runtime sert à exécuter ces données. 

Editor
  crée/modifie les scènes
        ↓
Fichiers de scène / prefabs / assets
        ↓
Runtime
  exécute le jeu

Cette séparation sera importante pour ton moteur : il faudra d’abord un cœur exécutable, puis progressivement des outils d’édition.

Comparaison simplifiée avec Godot

GodotUnity
Architecture centrée sur les Nodes.Architecture centrée sur GameObject + Components.
Un Node a souvent un rôle spécialisé.Un GameObject reçoit ses rôles via ses Components.
La scène est un arbre de Nodes.La scène est une hiérarchie de GameObjects.
GDScript est très intégré, même si C# existe.C# est central pour les scripts utilisateur.

Vers ton moteur C#

Les briques principales à reproduire, dans une version très simple, seront :

  • GameLoop : boucle principale du moteur ;
  • Scene : conteneur d’objets ;
  • GameObject : entité vide ;
  • Component : brique de comportement ;
  • Transform : position, rotation, taille ;
  • Renderer2D : affichage de sprites ;
  • Camera2D : point de vue sur le monde ;
  • ResourceLoader : chargement des ressources ;
  • Prefab : sauvegarde et réutilisation d’objets ;
  • Editor : interface pour inspecter et modifier la scène.
